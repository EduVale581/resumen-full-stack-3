main

Código en producción. Solo recibe merges de release y hotfix. Siempre estable.

develop

Integración continua. Base para features. Próxima versión en construcción.

feature/*

Una rama por funcionalidad. Sale de develop, vuelve a develop via PR.

release/*

Preparación de versión: bugfixes y docs. Merge a main y develop al cerrar.

hotfix/*

Corrección urgente en producción. Sale de main, merge a main y develop.

Flujo

GitFlow Paso a Paso

// PASO 01
Iniciar proyecto
git init
git flow init
# develop se crea automático
git checkout -b develop
// PASO 02
Nueva feature
git flow feature start login
# → crea feature/login
# ... desarrollas ...
git flow feature finish login
// PASO 03
Release
git flow release start 1.1.0
# bugfixes, docs, versión
git flow release finish 1.1.0
# → merge a main + develop
// PASO 04
Hotfix urgente
git flow hotfix start fix-auth
# corriges el bug crítico
git flow hotfix finish fix-auth
# → merge a main + develop

Errores

Errores Comunes de Hoy

🔥 Merge conflict sin resolver
CONFLICT (content): Merge conflict in src/auth.js
Automatic merge failed; fix conflicts
and then commit the result.

Dos ramas modificaron la misma línea. Ocurre al hacer merge o pull sin sincronizar develop primero.

✓ Solución git fetch origin
git rebase origin/develop
# Resolver conflictos en editor
git add . && git rebase --continue
💀 Push forzado en rama compartida
! [rejected] main → main (non-fast-forward)
hint: Updates were rejected because the tip
of your current branch is behind its remote.

Se usó git push --force en main o develop, sobreescribiendo el historial de otros desarrolladores.

✓ Solución segura # Nunca --force en ramas compartidas
git push --force-with-lease
# Solo si NADIE más hizo push
🗝️ Credenciales / secretos en commits
+ API_KEY=sk-prod-abc123xyz
+ DB_PASSWORD=supersecret2024
+ AWS_SECRET=AKIAIOSFODNN7EXAMPLE

Se commitean archivos .env o configs con secretos reales. Una vez en el historial, son públicos aunque se borren.

✓ Prevención # .gitignore
.env
.env.local
*.key
git secret / vault para secretos
⏪ Deshacer commit ya pusheado
fatal: You are in 'detached HEAD' state.
O peor: reset --hard borra trabajo del equipo

Usar git reset --hard en commits ya subidos al remoto destruye historial compartido.

✓ Usa revert en lugar de reset # Crea commit inverso (seguro)
git revert <commit-hash>
git push origin main
🐘 Archivos grandes en el repo
remote: error: File videos/demo.mp4 is
106.50 MB; exceeds GitHub's file size
limit of 100.00 MB

Binarios, videos, modelos ML o node_modules commiteados por accidente. Git no maneja bien archivos grandes.

✓ Solución # Usar Git LFS para binarios
git lfs track "*.mp4"
# .gitignore para node_modules
🌀 Rama desincronizada (diverged)
Your branch and 'origin/feature/login'
have diverged, and have 3 and 5 different
commits each, respectively.

La rama local y la remota avanzaron por separado. Ocurre al trabajar en la misma rama desde dos máquinas.

✓ Solución # Rebase interactivo para limpiar
git fetch origin
git rebase origin/feature/login
git push --force-with-lease
Pro Tips

Hábitos Esenciales

📝 Conventional Commits

Usa el estándar: feat:, fix:, chore:, docs:. Permite changelogs automáticos y semver.

🔍 Pull Request chico

PRs de menos de 400 líneas son revisados 2x más rápido. Un PR = una funcionalidad. Divide features grandes.

🛡️ Branch protection rules

Protege main y develop: requiere PR, mínimo 1 review y CI verde antes de hacer merge. Nunca commit directo.

🔄 Sincroniza seguido

Haz git pull --rebase de develop al menos una vez al día en features largas para evitar conflictos masivos.

🏷️ Tags semánticos

Etiqueta cada release con semver: v1.2.0. Permite rollbacks precisos y trazabilidad en producción.

🤖 Git Hooks

Usa pre-commit para lint y tests rápidos, y commit-msg para validar conventional commits automáticamente.
