// DESIGN_PATTERNS

Patrones de Diseño
en Microservicios

Soluciones probadas para problemas recurrentes en arquitecturas distribuidas

RESILIENCIA · 01 ⚡
Circuit Breaker

Detiene llamadas a un servicio fallido para evitar cascada de errores. Tiene estados: cerrado, abierto y semiabierto.

Estado: OPEN
→ Fallback activado
RESILIENCIA · 02 🔄
Retry Pattern

Reintenta automáticamente operaciones fallidas con backoff exponencial para evitar sobrecargar el servicio destino.

Intent 1 → falla
Intent 2 (2s) → falla
Intent 3 (4s) → ✓
RESILIENCIA · 03 🛡️
Bulkhead

Aísla recursos entre servicios (hilos, conexiones) para que el fallo de uno no consuma los recursos de otros.

Pool A: 10 threads
Pool B: 10 threads
→ Aislados
DATOS · 01 📝
Saga Pattern

Gestiona transacciones distribuidas como una serie de transacciones locales con compensaciones en caso de fallo.

Crea orden → Reserva
stock → Cobra pago
↩ Compensar si falla
DATOS · 02 🗂️
CQRS

Separa las operaciones de escritura (Commands) de las de lectura (Queries) usando modelos y bases de datos distintas.

Write DB → Events
↓
Read DB (optimized)
DATOS · 03 📡
Event Sourcing

El estado de la aplicación se determina reproduciendo una secuencia de eventos inmutables almacenados.

Evento1 + Evento2
+ Evento3 = Estado
actual reconstruido
COMUNICACIÓN · 01 🎭
Sidecar

Despliega capacidades auxiliares (logging, proxy, seguridad) como contenedor separado junto al servicio principal.

App Container +
Sidecar (Envoy)
→ Service Mesh
COMUNICACIÓN · 02 🔍
Service Discovery

Los servicios se registran y se localizan dinámicamente sin necesidad de IPs hardcodeadas. Client-side o server-side.

Service A → Registry
"¿Dónde está B?"
→ ip:port dinámico
COMUNICACIÓN · 03 🚦
Strangler Fig

Migra gradualmente un monolito a microservicios reemplazando funcionalidades una a una, sin big bang.

Monolito → 80%
Micro nuevo → 20%
→ Migración gradual
Resiliencia — fallos y tolerancia
Datos — consistencia y estado
Comunicación — integración y migración
